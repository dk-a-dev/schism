# Schism v1.1 — Design: Bill-Reading Harness, Claim Links, Expense UX, Fixes

Approved 2026-07-09. Four phases, shipped in order **D → C → A → B**; the user tests each phase on
the emulator/phone as we go. Guiding motto for the AI work: **small model, optimizations nobody
thinks of — optimize at every level. OCR is the eyes; the text model just has to be used better.**
(Reference: PennyWise AI runs tiny Gemma 3 1B and leans on a deterministic pipeline; we go further
with a constraint-verified harness.)

---

## Phase D — Bugs & clean slate

### D1. Stale balances/activity after editing an expense (confirmed root cause)
`GroupDetailViewModel.refresh()` runs only in `init` and after settle. Returning from the expense
editor reuses the ViewModel, so balances + activities (imperative fetches, not Room-observed) stay
stale, and the new UPDATE_EXPENSE activity never appears.
**Fix:** re-run `refresh()` on the group-detail screen's `ON_RESUME` (Lifecycle observer in the
composable) so any return — from edit, itemised, or settle — re-fetches. Keep it cheap: balances +
activities only (group/expenses are Room-observed and already fresh).
**Test:** ViewModel test that a second `refresh()` updates `balances`/`activities` state from a
changed fake API.

### D2. Prod data wipe for a clean retest
Extend the seed command: `WIPE_ALL=1` env truncates `users` and `groups` (cascades) and skips the
mock inserts unless `SEED=1` is also set. Run once against the deployed `DATABASE_URL` by the user.
No endpoint — this stays an operator command, never callable via HTTP.

### D3. Small-bug sweep
While in the area: expense date not editable in the editor (add a date field), group list pull-to-
refresh also refreshes balances-relevant caches, and any crashes found during emulator testing of
D/C.

## Phase C — Expense UX pack

- **C1. Delete expense** — visible only to the creator (same `addedBy` gate as edit) on the expense
  detail/editor; confirmation dialog; calls existing `DELETE /groups/{id}/expenses/{expenseId}`;
  activity feed already renders "removed".
- **C2. You-paid default** — Add Expense hides the payer dropdown; payer defaults to the active
  participant ("you"). A small "Paid by you · change" text button reveals the dropdown for the rare
  case someone else paid. Edit mode always shows the current payer.
- **C3. Scan-a-bill from Add Expense** — a receipt icon beside the Amount field opens the existing
  bill scan; on result the user picks: "fill amount only" (total into the form) or "switch to
  itemised split" (navigates with the parsed draft).
- **C4. Qty steppers** — replace tap-to-cycle share chips with explicit − / + steppers per
  participant per item (0 ↔ 1 ↔ 2 …, long-press not required). The item's own qty in the edit
  dialog also becomes a stepper. Fractional portions arrive with claim links (Phase B) — steppers
  here stay integer.
- **C5. Locked-expense breakdown viewer** — tapping any expense opens a read-only sheet: title,
  amount, date, payer, per-person shares (BY_AMOUNT shows exact amounts), notes; "Edit"/"Delete"
  buttons appear only for the creator. Replaces today's behavior where non-creators get nothing.
- **C6. Notes on itemised flow** — optional notes field before Create expense.
- **C7. Voice polish** — listening dialog shows live partial transcript (RecognitionListener
  partial results); after parsing, a confirm preview ("₹800 · Dinner · split with Riya, Sam —
  Apply?") instead of silently mutating the form.

## Phase A — Bill-reading engine v3: the harness

Flagship path stays **on-device OCR + small text LLM**, but the LLM is demoted from "reader" to
"classifier/fixer", and every result is verified by arithmetic. Components:

1. **Geometry pass** (`ReceiptScanner` v2): ML Kit boxes → visual rows (exists) **and column
   detection** (price column = X-aligned decimal cluster). Segment the **items region**: between
   the first price-bearing row (or an "Item/Qty/Amt" header) and the first totals row ("Item
   Total", "Sub Total"). Everything outside is metadata/fees.
2. **Source templates**: detect Swiggy / Zomato / Blinkit / paper-POS from keywords ("Bill
   Details", "Item Total", "Platform fee", "Order summary", "Covers", "GST@"). Templates contribute:
   known fee-line names (fees ≠ items ≠ taxes), "FREE"/strikethrough handling (two prices on a line
   → right-most is paid, e.g. Blinkit ₹96→₹72), pack-size vs qty ("2 x 80 g x 1" → qty 1, name
   keeps "Pack of 2"), and Swiggy/Zomato sub-option lines (indented "Cilantro Lime Rice") folding
   into the parent item's name.
3. **Candidate extraction** (deterministic): produce typed candidates — `{line, kind-guess:
   item|fee|tax|discount|metadata, name, qty, price(s), confidence}` — from the geometry +
   templates. This runs with no model and is unit-testable.
4. **LLM as classifier/fixer** (small model's easy job): prompt contains the candidate table (not
   raw OCR) in chunks of ~8 lines; the model returns per-line `{kind, cleanName, qty}` corrections
   only. Strict-JSON + repair-retry harness (exists) is reused. No amounts are ever produced by the
   model — amounts always come from OCR digits.
5. **Constraint solver (verification)**: assert `Σ items ≈ Item Total`, `Item Total + Σ fees +
   Σ taxes − Σ discounts ≈ Bill Total` (tolerance max(1%, ₹2)). On violation, search over candidate
   in/out toggles (bills have ≤ ~30 lines; bounded subset-sum with pruning) to find the assignment
   satisfying the bill's own totals; prefer the LLM labels, flip the fewest. Output carries a
   `verified: Boolean` the UI shows ("✓ totals verified" vs "check the items").
6. **Fees & taxes in the split**: fees/taxes/discounts land in the existing proportional
   `taxMinor`-style pot (extended to signed adjustments so discounts subtract).
7. **Model picker** (Settings → On-device AI): "Qwen2.5 1.5B (text, default)", "Qwen3-VL 2B
   (vision)", "Qwen3-VL 4B (vision, 8GB+ phones)", "Gemma 3n E2B (vision)". Vision entries feed the
   *image* directly to the model (llama.cpp runtime for Qwen3-VL GGUF; MediaPipe for Gemma 3n);
   their output flows through the SAME candidate/constraint verification. Backend `/model?variant=`
   serves/redirects each; existing download worker gains variant awareness.
8. **Golden fixtures**: the user's real screenshots (POSIST paper bill, 2× Swiggy, Zomato/
   Theobroma, Blinkit) captured as OCR-line fixtures + exact expected drafts; JUnit regression
   suite gates all parser/harness changes. New formats get added as fixtures when found.

Note on scope: llama.cpp integration (A7 vision entries) is the largest single lift and lands last
within Phase A; the harness (A1–A6, A8) works with the existing MediaPipe text model immediately.

## Phase B — Claim links (in-app)

Flow: scan bill → "Split by claims" → share one link → everyone in the group claims what they had →
creator finalizes → expense created.

- **Backend**: `claim_sessions` (id, group_id, creator participant, title, currency, items JSONB
  snapshot `{name, qty, amountMinor}`, tax/fees minor, status open|finalized, created_at) and
  `claims` (session_id, item_index, participant_id, portion NUMERIC — 0.5 allowed). Endpoints:
  `POST /v1/groups/{id}/claim-sessions`, `GET /v1/claim-sessions/{sid}`,
  `PUT /v1/claim-sessions/{sid}/claims` (the caller's claims, idempotent replace),
  `POST /v1/claim-sessions/{sid}/finalize` (creator only → builds the BY_AMOUNT expense server-side
  with the same weighted math; unclaimed remainder goes to the creator or is split evenly — flag on
  finalize). Auth: must be a group member (linked user).
- **App**: "Split by claims" button on the itemised screen; a Claim screen (deep link
  `https://<backend>/c/<sid>` bouncing to `schism://claim/<sid>` like invites) with portion
  steppers (+/− in 0.5 steps), live "your share ₹…" and a per-item claimed/total indicator; the
  creator additionally sees per-person totals and the Finalize button. Screen polls `GET` every few
  seconds while visible (no websockets in v1).
- The finalized expense appears like any itemised expense (locked to creator, viewable breakdown
  via C5).

## Cross-cutting

- **Testing**: unit tests per component (candidates, templates, solver, claim math), golden
  fixtures for the harness, ViewModel tests for D1/C2/C4, backend httptest for claim endpoints.
  Emulator walkthrough after each phase (user-driven).
- **Releases**: tag at phase boundaries (D+C → 1.0.4, A → 1.1.0, B → 1.2.0), signed APK attached;
  backend redeploy required at A (model variants) and B (claim tables — migrations auto-run).
- **Out of scope for v1.1**: web claim pages (in-app only), websockets/live push, cloud AI,
  fractional portions outside claim links, iOS.
